ControlID Parameter (Window Control Identifiers)

Some commands have a ControlID parameter used to identify which control in a window to operate on. This parameter can be the text of the control or any other identifier described on this page.

Control Identifiers

The following identifiers are ordered by descending precedence. This detail is important in rare cases of ambiguity. For example, if a window has two edit controls, the first with the text "Edit2" and the second with ClassNN "Edit2", and Control, Hide,, Edit2 is used, the second control will be hidden, because ClassNN has higher precedence.

HWND

A control's HWND (short for handle to window) uniquely identifies that control for as long as it exists. To operate upon it, leave the ControlID parameter blank and specify ahk_id %ControlHwnd% for the WinTitle parameter (this also works on hidden controls even when DetectHiddenWindows is turned off). [v1.1.04+]: GuiControl and GuiControlGet allow the HWND to be specified directly in ControlID.

A control's HWND can be retrieved via DllCall(), in [v1.0.43.06+] via ControlGet Hwnd, WinGet ControlListHwnd, and MouseGetPos, in [v1.0.46.01+] via the Gui Add's Hwnd option, and in [v1.0.46.16+] via GuiControlGet Hwnd.

A HWND is especially useful when sending messages directly to a control via PostMessage, SendMessage, or when calling native APIs with DllCall(). It is also helpful when controls are dynamically created or reordered, since the HWND remains constant for the lifetime of the control.

Associated Variable

A GUI control may be identified by the name of the variable associated with it. This applies to controls created with Gui Add when the V option is used (e.g. vMyEdit). The variable name then serves as the control's identifier for use with GuiControl and GuiControlGet.

Using an associated variable is often the most convenient way to identify a GUI control, even for controls that are not input-capable (such as Text or GroupBox). In these cases, the variable does not store input, but still provides a stable and readable identifier.

ClassNN

A control's ClassNN is the name of its window class followed by its sequence number within the top-level window which contains it. The sequence number reflects the control's position among controls of the same class, in the order they were created. For example, Edit1 is the first edit control in a window and Button12 is the twelfth button.

Some class names include digits which are not part of the control's sequence number. For example, SysListView321 is the window's first ListView control, not its 321st. To retrieve the class name without the sequence number, pass the control's HWND to WinGetClass.

A control's ClassNN can be determined via Window Spy. Some commands such as ControlGetFocus, GuiControlGet Focus, WinGet ControlList, and MouseGetPos retrieve a ClassNN.

The sequence number is not a persistent identifier. It may change if the application recreates controls, changes their order, or uses dynamic or virtualized UI elements. Some modern frameworks (such as WPF or Electron) may not expose traditional Win32 class names at all, or may reuse generic class names for multiple unrelated controls.

ClassNN is often convenient when controls do not have unique text and when the control order is stable. When stability cannot be guaranteed, consider using a HWND or another identifier.

Text

The text of a control may be used as a identifier, e.g. OK for an OK button. The matching behavior is determined by SetTitleMatchMode. Use text when it is unique and stable; many controls update it dynamically or do not expose meaningful text.

A control's text can be determined via Window Spy. Some commands such as ControlGetText and GuiControlGet retrieve such text.

Omitted

For non-GUI commands, omitting the ControlID parameter causes them to operate on the target window's topmost control. However, a few of them have special behavior when omitting this parameter:

  • ControlClick clicks the target window itself if there are no controls.
  • ControlSend sends the keystrokes to the target window itself if there are no controls.
  • PostMessage and SendMessage send the message directly to the target window.

For GuiControlGet, omitting the ControlID parameter causes it to behave as though the control's associated variable name (if any) was specified.

Remarks

The following commands have a ControlID parameter: Control, ControlClick, ControlFocus, ControlGet, ControlGetPos, ControlGetText, ControlMove, ControlSend, ControlSetText, GuiControl, GuiControlGet, PostMessage, SendMessage. Most of these commands have examples on their corresponding page that demonstrate the use of control identifiers.

ClassNN and control text may change when applications recreate controls, alter their order, use dynamic or virtualized interfaces, or rely on frameworks that do not expose stable Win32 class names or text. When identifiers are unreliable, prefer using a HWND for lifetime-stable addressing or combine multiple criteria. As general guidance: use control text when it is unique and stable, use ClassNN when the control order is consistent and readability is desired, and use a HWND when maximum stability is required. Avoid hardcoding identifiers across application versions unless you control the target application.
